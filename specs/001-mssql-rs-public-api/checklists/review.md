# Spec & Plan Quality Gate Checklist: mssql-rs Public API Crate

**Purpose**: Thorough gate — validate completeness, clarity, consistency, and measurability of all requirements in spec.md and plan.md before implementation  
**Created**: 2026-03-05  
**Feature**: [spec.md](../spec.md) | [plan.md](../plan.md

**Depth**: Thorough gate (every FR checked individually)  
**Focus**: Full coverage — completeness, clarity, consistency, measurability, edge cases, API contract alignment, plan feasibility  
**Audience**: Reviewer (PR / pre-implementation gate)

## Requirement Completeness

- [x] CHK001 - Are ALL connection string keys that the crate will accept enumerated in FR-001 or in a referenced key table? The spec lists 7 keys; the plan/research maps 13. Six keys (Application Name, Packet Size, Application Intent, MultiSubnetFailover, Authentication, Access Token) are only documented in research.md, not in the spec. [Gap, Spec §FR-001] *(Resolved: FR-001 updated to list all 12 standard keys + unknown-key error + brace-quoting rule)*
- [x] CHK002 - Is FR-002 specific enough about what "returning results" means? The FR says "executing ad-hoc SQL queries… and returning results" without defining the return type, result shape, or behavior for DDL statements that produce no rows. [Clarity, Spec §FR-002] *(Resolved: FR-002 updated — DDL returns empty ResultSet with zero rows and no metadata)*
- [x] CHK003 - Does FR-005 enumerate all Value variants that will be in the public API? The spec says the enum must cover "integers, floats, strings, binary, date/time, decimal, UUID, XML, bit/bool, and NULL" (10 categories) but the data model defines 13 variants including Json, Vector, and a separate DateTime struct. Are Json and Vector in scope per the spec? [Gap, Spec §FR-005] *(Resolved: FR-005 updated to include JSON, vector, and clarified DateTime as a single struct)*
- [x] CHK004 - Is the DateTime representation specified as a requirement? The spec entity "Value" lists `DateTime(...)` with an ellipsis but never defines the struct fields or which temporal variants it subsumes. The DateTime struct definition only appears in the data model. [Gap, Spec §FR-005 / Entity: Value] *(Resolved: FR-005 now specifies all 8 DateTime fields with Option types and their semantics)*
- [x] CHK005 - Is FR-009 specific about parameter naming convention? The spec says "parameter values are sent separately" but does not specify whether parameters are named (`@p1`) or positional. The contract uses `&[(&str, Value)]` (named), which is an implementation decision not captured in the spec. [Clarity, Spec §FR-009] *(Resolved: FR-009 now specifies named parameters with `@` prefix via sp_executesql)*
- [x] CHK006 - Is FR-013 query timeout configuration fully specified? The FR says "connection timeout and query timeout configuration" but no spec requirement, user story, or acceptance scenario describes how query timeout is set (method parameter, connection string key, or Client method). The contract API has no query timeout surface. [Gap, Spec §FR-013] *(Resolved: FR-013 updated — both timeouts via connection string keys in seconds, query timeout cancels and returns Error::Timeout)*
- [x] CHK007 - Is the API surface for FR-017 (binary chunk streaming) defined in the spec? FR-017 specifies behavior ("yields data in chunks via `futures::Stream<Item = Result<Bytes>>`") but no spec entity, user story scenario, or FR defines the method used to open the binary stream on a Row or column. [Gap, Spec §FR-017] *(Resolved: FR-017 now specifies `open_binary_stream(index)` on Row as the entry point method)*
- [x] CHK008 - Are requirements for executing queries within an active transaction explicitly specified? US7 covers begin/commit/rollback/isolation but no FR requires executing queries or preparing statements within a transaction scope. The contract adds `Transaction::query()` and `Transaction::prepare()`, which have no spec-level backing. [Gap, Spec §US7 / §FR-016] *(Resolved: FR-018 added — query, query_with_params, and prepare within transaction scope)*
- [x] CHK009 - Are error behaviors for invalid/unknown connection string keys specified? FR-001 says unknown keys — the research says "produce a descriptive error" but the spec has no requirement for this. Behavior for duplicate keys, empty values, and whitespace-only values is also undefined. [Gap, Spec §FR-001] *(Resolved: FR-001 now requires unknown keys MUST produce an error)*
- [x] CHK010 - Is FR-006 exhaustive about which "standard Rust types" have built-in conversion trait implementations? The FR says "`i32`, `i64`, `f64`, `String`, `bool`, `Vec<u8>`, etc." — the "etc." is ambiguous. The contract adds `i8`, `i16`, `u8`, `u16`, `u32`, `f32`, `BigDecimal`, `uuid::Uuid`, `DateTime`. Are all of these spec requirements? [Ambiguity, Spec §FR-006] *(Resolved: FR-006 now enumerates all 16 built-in FromValue implementations explicitly — no "etc.")*
- [x] CHK011 - Is FR-011 complete regarding which metadata fields are exposed? The FR says "name, data type, nullability" — the ColumnMetadata entity in spec also lists "collation", which the FR omits. Are all four fields required? [Consistency, Spec §FR-011 vs Entity: ColumnMetadata] *(Resolved: FR-011 now lists all 4 fields including collation as `Option<String>`)*
- [x] CHK012 - Are requirements for savepoints within transactions specified? `mssql-tds` supports `save_transaction()` but neither the spec FRs nor US7 mention savepoints. Is this an intentional exclusion or a gap? [Gap, Spec §FR-016] *(Resolved: Savepoints explicitly listed as out of scope in Assumptions)*
- [x] CHK013 - Is a requirement for closing/disconnecting the Client explicitly specified? The contract defines `Client::close(self) -> Result<()>` but no FR or user story covers connection teardown behavior, error handling on close, or resource cleanup guarantees. [Gap] *(Resolved: Client entity now documents `close(self)` for explicit teardown; Drop also cleans up)*

## Requirement Clarity

- [x] CHK014 - Is "a small set of temporal variants" in FR-005 quantified? The spec uses this phrase without defining how many or which variants. The plan resolved this to a single DateTime struct, but the spec itself is ambiguous. [Ambiguity, Spec §FR-005] *(Resolved: FR-005 now says "a single DateTime struct" instead of "a small set of temporal variants")*
- [x] CHK015 - Is "descriptive error" in US1 scenario 2 defined with specific content requirements? The scenario says "a descriptive error indicating which part of the connection string is invalid" but doesn't specify the error variant, whether the offending key/value is included, or the error message format. [Ambiguity, Spec §US1] *(Resolved: US1.2 now specifies `Error::ConnectionStringInvalid` with a message identifying the offending key or value)*
- [x] CHK016 - Is "constant memory overhead" in SC-002 quantified with a measurable threshold? SC-002 says "memory usage does not grow proportionally to row count" but doesn't define an acceptable constant (e.g., <10 MB overhead regardless of row count) or measurement methodology. [Measurability, Spec §SC-002] *(Resolved: SC-002 now specifies RSS delta < 10 MB between 1K and 100K rows)*
- [x] CHK017 - Is SC-006 ("< 5% latency regression") defined with a measurement methodology? The criterion doesn't specify which operations are measured, workload profile, warm-up requirements, or statistical significance criteria. [Measurability, Spec §SC-006] *(Resolved: SC-006 now specifies `cargo bench` on 10K rows, 100 iterations, median latency; noted as design target not CI gate)*
- [x] CHK018 - Is "properly drained or the connection is left in a valid state for reuse" in US2 scenario 3 unambiguous? This is an "or" condition — which behavior should the implementation choose? The plan doesn't resolve this ambiguity. [Ambiguity, Spec §US2] *(Resolved: US2.3 now specifies rows are drained; the "or" ambiguity is removed)*
- [x] CHK019 - Are the Row access mode constraints (sequential vs random-access mutual exclusivity) specified in the spec? The data model and contract define two mutually exclusive access patterns, but the spec entity "Row" only says "yields column values one at a time… Provides indexed and trait-based access" without addressing the interaction between these modes. [Gap, Spec Entity: Row] *(Resolved: Row entity now describes two mutually exclusive access modes — random-access via `get()`/`value()` and sequential via `next_column()` — with the first call determining the mode)*

## Requirement Consistency

- [x] CHK020 - Does FR-017's "MUST support" language conflict with the plan's decision to defer binary chunk streaming (R4)? The spec uses RFC-2119 "MUST" language, implying the initial release must include this capability. The plan explicitly defers it to a follow-up iteration. This needs reconciliation — either downgrade the spec to "SHOULD" or include the feature in scope. [Conflict, Spec §FR-017 vs Plan §R4] *(Resolved: FR-017 downgraded to SHOULD, with note that initial impl MAY buffer internally and true PLP streaming is deferred per R4)*
- [x] CHK021 - Is US2 acceptance scenario 5 (10 MB varbinary streaming) achievable if FR-017 is deferred? The scenario explicitly requires "data is yielded in chunks without buffering the entire 10 MB in memory", which contradicts the plan's deferred-buffering approach. [Conflict, Spec §US2.5 vs Plan §R4] *(Resolved: US2.5 updated — data yielded in chunks, but initial impl MAY buffer internally with synthetic chunks; note references FR-017)*
- [x] CHK022 - Are the Value variant choices consistent between spec and data model? The spec entity lists ~10 categories; the data model defines 13 variants. Specifically, `Json(String)` and `Vector(Vec<f32>)` appear in the data model but are not mentioned in the spec's FR-005 category list or Value entity description. [Consistency, Spec §FR-005 vs data-model] *(Resolved: FR-005 and Value entity both updated to include Json(String) and Vector(Vec<f32>))*
- [x] CHK023 - Do column-level streaming requirements (FR-003b) have a corresponding method in the public API contract? The contract's Row type exposes `get()`, `get_by_name()`, `value()`, `len()`, `is_empty()`, `metadata()`, and `into_values()` — but no `next_column()` method. The data model defines `next_column()` for sequential streaming, but the contract omits it. [Consistency, Spec §FR-003 vs contract] *(Resolved: next_column(&mut self) -> Result<Option<Value>> added to Row contract with mutually-exclusive mode note)*
- [x] CHK024 - Is FR-011 metadata scope consistent with the ColumnMetadata entity? FR-011 requires "name, data type, nullability" (3 fields); the entity adds "collation" (4 fields). The contract struct has all 4. Which is authoritative? [Consistency, Spec §FR-011 vs Entity: ColumnMetadata] *(Resolved: FR-011 now lists all 4 fields including collation; spec is authoritative)*
- [x] CHK025 - Are Transaction method requirements consistent between spec and contract? US7 specifies begin/commit/rollback/isolation-level. The contract adds `Transaction::query()`, `Transaction::query_with_params()`, and `Transaction::prepare()` — executing queries within a transaction is not explicitly required by any FR. [Consistency, Spec §US7 vs contract] *(Resolved: FR-018 added to spec backing Transaction::query, query_with_params, and prepare)*
- [x] CHK026 - Is the deferred cleanup pattern (R7, R8) consistent with US7 scenario 4? US7.4 says the transaction is "silently rolled back" on drop, implying immediate rollback. The plan uses deferred rollback (flag set on drop, actual rollback on next operation). If no subsequent operation occurs (Client is also dropped), the rollback never executes. Is this acceptable? [Consistency, Spec §US7.4 vs Plan §R7] *(Resolved: US7.4 now specifies deferred rollback with no blocking I/O in drop; server reclaims on connection close if no subsequent operation)*

## Acceptance Criteria Quality

- [x] CHK027 - Are US5 acceptance scenarios sufficient for parameter edge cases? Only 2 scenarios are defined (correct execution + injection safety). Missing: wrong parameter count, missing parameter name, parameter type mismatch with column type, null parameter values. [Coverage, Spec §US5] *(Resolved: US5 now has 4 scenarios — added parameter count mismatch and NULL parameter handling)*
- [x] CHK028 - Does SC-001 ("under 10 lines of application code") have a clear counting methodology? Should imports, `#[tokio::main]`, error propagation (`?`), and `use` statements be counted? The quickstart examples vary in line count depending on interpretation. [Measurability, Spec §SC-001] *(Resolved: SC-001 now excludes `use` imports, `#[tokio::main]`, `fn main()` boilerplate; `?` counts on same line)*
- [x] CHK029 - Is SC-003 specific about how "all supported SQL Server data types" are enumerated and validated? The criterion lists example types but uses "etc." — no definitive type matrix is provided. [Measurability, Spec §SC-003] *(Resolved: SC-003 now enumerates all 28 SQL Server types explicitly — no "etc.")*
- [x] CHK030 - Does SC-005 ("doc-tests") specify which doc-tests must compile and pass — all public items, or just key items? [Clarity, Spec §SC-005] *(Resolved: SC-005 now scoped to all re-exports in contracts/public-api.md plus crate-level `//!` doc)*

## Scenario Coverage

- [x] CHK031 - Are requirements for DDL statement execution (CREATE TABLE, ALTER, DROP) and their result handling defined? FR-002 says "executing ad-hoc SQL queries and returning results" but DDL produces no result set. What does `Client::query("CREATE TABLE ...")` return? [Coverage, Gap] *(Resolved: FR-002 updated — DDL returns empty ResultSet with zero rows and no metadata)*
- [x] CHK032 - Are concurrent operation requirements defined? The plan enforces single-active-operation via `&mut` borrows, but the spec doesn't specify whether `Client` is `Send`/`Sync` or whether `cancel()` (which takes `&self`) can be called from another thread/task. [Gap] *(Resolved: Assumptions now specify Client is Send but not Sync; cancel() via CancelHandle is thread-safe)*
- [x] CHK033 - Are requirements for connection reconnection/recovery after I/O failure defined? Edge case 1 says the client should be in a "state where the developer can reconnect" but no FR specifies a reconnect method or behavior. [Gap, Spec Edge Cases] *(Resolved: Assumptions now explicitly state reconnection is out of scope; new Client required after I/O error)*
- [x] CHK034 - Are requirements for server-initiated connection termination (e.g., server kill, idle timeout) specified? [Gap] *(Resolved: Edge cases now cover server-initiated disconnection returning Error::Io; Assumptions cover reconnection policy)*
- [x] CHK035 - Are requirements for zero-column result sets or metadata-only responses specified? [Gap] *(Resolved: Edge cases now cover zero-column result sets — empty metadata vec, zero rows returned)*
- [x] CHK036 - Are requirements for handling SQL Server informational messages (severity < 10) specified? These are not errors but may contain useful warnings. No spec requirement addresses them. [Gap] *(Resolved: Assumptions now state informational messages (severity < 10) are not surfaced; deferred to future iteration)*

## Edge Case Coverage

- [x] CHK037 - Are requirements for integer overflow during FromValue narrowing conversions (e.g., `Value::Int(i64::MAX)` extracted as `i32`) specified? The data model says "with range check for narrowing" but no spec requirement defines the error behavior. [Edge Case, Gap] *(Resolved: Edge cases now specify `Error::TypeConversion` for range violations in narrowing conversions)*
- [x] CHK038 - Are requirements for non-UTF-8 string encoding edge cases fully specified? Edge case 3 mentions "collation-aware strings" but doesn't define behavior when a non-Unicode collation produces bytes that cannot be represented as UTF-8 in `Value::String(String)`. [Edge Case, Spec Edge Cases §3] *(Resolved: Edge cases now specify mssql-tds handles encoding via encoding_rs; Value::String is always valid UTF-8 with U+FFFD replacement for unmappable bytes)*
- [x] CHK039 - Are requirements for deferred cleanup failure defined? If `pending_rollback` or `pending_unprepare` encounters a server error during execution on the next operation, what error does the caller see — the cleanup error or their intended operation's error? [Edge Case, Gap] *(Resolved: Edge cases now specify cleanup error is returned from next operation; the intended operation does not execute)*
- [x] CHK040 - Are requirements for connection string values containing special characters (semicolons, equals signs, braces) defined? Research R1 mentions brace-wrapping per ODBC spec, but no spec requirement or edge case covers this. [Edge Case, Gap] *(Resolved: FR-001 already requires brace-wrapping for values containing semicolons or equals signs per ODBC spec)*
- [x] CHK041 - Are requirements for handling a dropped ResultSet mid-iteration when queries remain on the wire specified? US2.3 says "properly drained or valid state for reuse" — but what is the cost of draining (blocking until all rows received)? Is a connection reset acceptable as an alternative? [Edge Case, Spec §US2.3] *(Resolved: Edge cases now specify ResultSet drop drains remaining rows to keep Client usable; draining is mandatory, connection reset not used)*

## Non-Functional Requirements

- [x] CHK042 - Are thread-safety requirements (`Send`/`Sync` bounds) for all public types explicitly specified? The contract doesn't annotate any type with `Send + Sync` requirements, but async usage across `.await` points requires `Send`. [Gap] *(Resolved: Assumptions now specify Client is Send but not Sync; `&mut self` enforces single-operation access)*
- [x] CHK043 - Are crate-level documentation requirements beyond SC-005 specified? No requirement covers crate-level doc comments (`//! ...` in lib.rs), a README.md for the crate, or example programs in `examples/` directory. [Gap] *(Resolved: SC-005 now requires crate-level doc; Assumptions note README/examples not required for initial release)*
- [x] CHK044 - Are minimum supported Rust version (MSRV) requirements for the new crate specified? The plan says Rust 1.90 / Edition 2024 but no spec requirement establishes an MSRV policy. [Gap] *(Resolved: Assumptions now state MSRV matches workspace: Rust 1.90, Edition 2024)*

## Dependencies & Assumptions

- [x] CHK045 - Are the `bigdecimal` and `uuid` crate dependencies that leak into the public API specified as requirements? `Value::Decimal(BigDecimal)` and `Value::Uuid(uuid::Uuid)` expose third-party types in the public API, coupling `mssql-rs`'s semver to these crates. No spec requirement acknowledges this trade-off. [Assumption, Gap] *(Resolved: Assumptions now acknowledge BigDecimal/Uuid semver coupling as accepted trade-off with re-export)*
- [x] CHK046 - Is assumption 3 ("async-first usage; synchronous blocking wrapper is out of scope") validated against the target user base? If any consumer needs sync access, this is a hard exclusion that should be explicitly acknowledged as a non-goal in the spec. [Assumption, Spec §Assumptions.3] *(Resolved: Assumptions now clarify async-first is intentional; target users are async Rust/Tokio apps; sync wrapper deferred)*
- [x] CHK047 - Is the assumption that `mssql-tds` internal types (`ColumnValues`, `ClientContext`) will remain accessible to workspace-internal consumers validated? If `mssql-tds` makes these types fully private, the wrapping approach breaks. [Assumption] *(Resolved: Assumptions now document dependency on mssql-tds pub(crate) visibility; note to revisit if mssql-tds changes visibility)*

FR-017 (binary streaming) — downgraded to SHOULD, deferred per R4. FR-018 — Transaction-scoped queries added. [Traceability, Spec §FR-013, §FR-017, §FR-018 vs contract] *(Resolved: FR-013 clarified, FR-017 aligned with plan, FR-018 added)*

R4 (PLP deferral) contradicts FR-017 MUST language — is this an acknowledged deviation? [Traceability] *(Resolved: All decisions traceable — R1→FR-001, R2→FR-005, R3→FR-003, R4→FR-017 (aligned: SHOULD), R5→FR-012, R6→FR-003, R7→FR-016/US7.4, R8→FR-015)*

[Consistency, contract §Row vs §FromValue] *(Resolved: Assumptions now document FromValue takes owned Value; Row::get clones internally; negligible cost for non-Binary variants)*
